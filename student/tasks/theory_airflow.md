# Теория: Airflow — зачем он нужен и как работает

## Проблема

Представьте: вы аналитик, и каждое утро вам нужно:

1. Посчитать, сколько новых пользователей зарегистрировалось вчера
2. Посчитать выручку за вчера
3. Обновить таблицу `daily_metrics`
4. Проверить, что данные корректные

Вы можете делать это руками — каждое утро открывать DataGrip и запускать SQL. Но что если вы заболели? Уехали в отпуск? Или у вас 50 таких отчётов?

В реальных компаниях эти задачи **автоматизируют**. Именно для этого нужен Airflow.

---

## Что такое Airflow?

**Apache Airflow** — это инструмент для автоматизации и планирования задач с данными. Его создали в Airbnb в 2014 году, и сейчас его используют почти все крупные компании: Яндекс, Тинькофф, Ozon, Авито, Google, Uber, Netflix.

Простыми словами: вы описываете **что нужно сделать** и **в каком порядке**, а Airflow **запускает это по расписанию** и следит, чтобы всё выполнилось.

---

## DAG — основное понятие

**DAG** (Directed Acyclic Graph) — это набор задач с зависимостями. "Направленный ациклический граф" звучит страшно, но на практике это просто:

```
Создать таблицу → Посчитать метрики → Проверить результат
```

Каждая стрелка — зависимость. Airflow гарантирует, что:
- "Посчитать метрики" не запустится, пока "Создать таблицу" не выполнится
- Если "Посчитать метрики" упадёт с ошибкой — "Проверить результат" не запустится
- Вы увидите в UI, какая именно задача сломалась и почему

---

## Как это выглядит в коде?

DAG — это обычный Python-файл. Вот минимальный пример:

```python
from airflow import DAG
from airflow.providers.postgres.operators.postgres import PostgresOperator
from datetime import datetime

with DAG(
    dag_id="my_first_dag",
    schedule_interval="@daily",    # Запускать каждый день
    start_date=datetime(2024, 1, 1),
    catchup=False,
) as dag:

    task_1 = PostgresOperator(
        task_id="count_users",
        postgres_conn_id="mentoring_db",
        sql="SELECT count(*) FROM shared_data.users;",
    )
```

Вы кладёте этот файл в папку `dags/`, и Airflow автоматически подхватывает его через 30-60 секунд.

## Ключевые понятия

### Operators — типы задач
- **PostgresOperator** — выполняет SQL-запрос в базе данных
- **PythonOperator** — запускает Python-функцию
- **BashOperator** — выполняет команду в терминале

### Connections — подключения
Airflow хранит настройки подключения к базам данных, API и другим сервисам. В нашей среде уже настроено подключение `mentoring_db` — оно указывает на вашу локальную PostgreSQL.

### Schedule — расписание
- `@daily` — каждый день в полночь
- `@hourly` — каждый час
- `0 9 * * *` — каждый день в 9:00 (cron-формат)

### Идемпотентность
Это важнейший принцип: DAG можно запустить повторно, и результат будет тот же. Если вчерашние метрики уже посчитаны и вы запустите DAG ещё раз — данные обновятся, но не задублируются. Для этого используют `ON CONFLICT ... DO UPDATE` в SQL.

## Зачем это аналитику?

| Без Airflow | С Airflow |
|-------------|-----------|
| Руками запускаете SQL каждое утро | Запускается автоматически |
| Забыли запустить — данных нет | Работает даже когда вы спите |
| Ошибка — никто не узнал | Ошибка — Airflow покажет в UI, отправит алерт |
| 50 отчётов = 50 ручных запусков | 50 DAG'ов работают параллельно |
| Непонятно, что от чего зависит | Граф зависимостей виден визуально |

## Airflow UI

Откройте http://localhost:8080 — это веб-интерфейс Airflow. Здесь вы можете:

- Видеть все DAG'и и их статус
- Запускать DAG вручную (кнопка Play)
- Смотреть граф задач (вкладка Graph)
- Читать логи каждой задачи
- Видеть историю запусков

## Как это работает в реальных компаниях?

В крупных компаниях Airflow — центральный инструмент дата-команды:

- **Аналитики** пишут DAG'и, которые считают метрики и обновляют витрины
- **Дата-инженеры** строят сложные пайплайны: загрузка данных из внешних источников, трансформации, выгрузки
- **ML-инженеры** используют Airflow для обучения и деплоя моделей

Типичный день аналитика с Airflow:
1. Утром приходишь на работу
2. Открываешь Airflow UI — проверяешь, все ли DAG'и отработали за ночь
3. Если что-то красное (failed) — смотришь логи, чинишь
4. Пишешь новый DAG или правишь существующий
5. Делаешь pull request, проходишь код-ревью
6. После мержа в main — DAG автоматически подхватывается
