# Теория: Git и GitHub — зачем это аналитику

## Проблема

Вы написали DAG, он работает. Потом вы решили его "улучшить", поменяли SQL-запрос — и всё сломалось. А старую версию вы не сохранили. Или: вы работаете вдвоём над одним проектом и оба редактируете один файл — чьи изменения победят?

Именно для решения этих проблем существует **Git**.

---

## Что такое Git?

**Git** — это система контроля версий. Она сохраняет историю всех изменений в вашем коде. Каждое сохранение называется **commit** (коммит).

Представьте Google Docs, но для кода:
- Вы видите, кто, когда и что изменил
- Можете откатиться к любой прошлой версии
- Несколько человек могут работать параллельно без конфликтов

Git используют **все** разработчики и аналитики в IT-компаниях. Без исключений.

---

## Что такое GitHub?

**GitHub** — это облачный хостинг для Git-репозиториев. Место, где хранится ваш код в интернете.

- **Git** = инструмент на вашем компьютере (локально)
- **GitHub** = сайт, где код доступен всей команде (удалённо)

Аналогия: Git — это Word на вашем компьютере, GitHub — это Google Drive, куда вы загружаете документы, чтобы коллеги их видели.

---

## Основные команды

### Ежедневный рабочий цикл

```bash
# 1. Получить свежий код из GitHub
git pull

# 2. Создать новую ветку для вашей задачи
git checkout -b feature/daily-metrics-dag

# 3. Работаете: пишете код, редактируете файлы...

# 4. Посмотреть, что изменилось
git status

# 5. Добавить файлы в коммит
git add dags/my_new_dag.py

# 6. Сохранить изменения (коммит)
git commit -m "Add daily metrics DAG"

# 7. Отправить на GitHub
git push origin feature/daily-metrics-dag
```

### Что означает каждая команда

| Команда | Что делает | Аналогия |
|---------|-----------|----------|
| `git clone` | Скачать проект с GitHub | Скачать папку из облака |
| `git pull` | Обновить локальную копию | Синхронизировать Google Drive |
| `git status` | Показать изменённые файлы | Посмотреть, что изменилось |
| `git add` | Подготовить файлы к сохранению | Выбрать файлы для отправки |
| `git commit` | Сохранить изменения локально | Нажать "Сохранить" |
| `git push` | Отправить на GitHub | Загрузить в облако |

---

## Ветки (branches)

Ветка — это параллельная копия кода. Вы работаете в своей ветке и не ломаете чужой код.

```
main ──────────────────────────────── основная ветка
  \                                /
   feature/daily-metrics ────────── ваша ветка (merge)
```

- **main** — основная ветка, здесь лежит стабильный код
- **feature/...** — ваша ветка, где вы работаете над задачей
- Когда задача готова — вы делаете **merge** (слияние) обратно в main

---

## Pull Request (Merge Request)

**Pull Request** (PR) — это запрос на слияние вашей ветки в main. Это ключевой процесс в любой IT-компании:

1. Вы написали код в своей ветке
2. Создаёте PR на GitHub
3. Ментор (или коллега) **просматривает ваш код** — это называется **код-ревью**
4. Оставляет комментарии: "Здесь лучше сделать так", "А тут забыл обработать ошибку"
5. Вы исправляете замечания
6. Ментор одобряет (approve)
7. Вы нажимаете **Merge** — код попадает в main

### Зачем код-ревью?

- Другой человек находит ошибки, которые вы не заметили
- Вы учитесь на замечаниях
- Команда знает, какие изменения вносятся
- Снижается риск сломать что-то в продакшене

---

## Зачем Git аналитику?

| Без Git | С Git |
|---------|-------|
| "Финальная_версия_v2_ФИНАЛ(1).py" | Чистая история всех версий |
| Сломал код — не откатить | `git checkout` на прошлую версию |
| Отправил файл по почте | Ссылка на GitHub, всё в одном месте |
| Непонятно, что менялось | `git diff` покажет каждую строку |
| Коллега перезаписал вашу работу | Ветки и merge решают конфликты |

---

## Как это работает в реальных компаниях?

В большой компании **весь код проходит через Git**:

- Аналитик пишет DAG → push → PR → код-ревью → merge → Airflow подхватывает
- Никто не деплоит код напрямую — только через PR
- У каждой задачи своя ветка
- Есть CI/CD (автоматические проверки): код проходит тесты перед merge

Это касается не только разработчиков — **аналитики, дата-инженеры, ML-инженеры** — все работают через Git.

## Как создать Pull Request

После того как вы написали DAG и запушили его:

```bash
git push origin feature/daily-metrics-dag
```

1. Откройте GitHub: https://github.com/simulator-mentoring/data-platform
2. Появится баннер "Compare & pull request" — нажмите на него
3. Или: вкладка **Pull requests** → **New pull request**
4. Выберите вашу ветку → нажмите **Create pull request**
5. Напишите описание: что сделали и зачем
6. Отправьте ментору на ревью

---

## Итого

Git и GitHub — это не "инструмент для программистов". Это стандарт работы в любой дата-команде. Если вы хотите работать аналитиком в IT — вам нужно уметь:

- Создавать ветки и коммитить код
- Делать pull request'ы
- Проходить код-ревью
- Работать с `git pull` / `git push`
